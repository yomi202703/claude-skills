Stand up the review server for a no-ground-truth judgment loop: one config-driven developer diagnostic/evaluation server, plus a human GT-creation surface (a mode of it, or a separate clean handover deliverable — S1). Domain-agnostic: vocabulary, axes, unit, and questions are injected from config, never hard-coded. Start from `template/` (runnable skeleton, stdlib only, gates enforced in store.py + server.py); adapt only `contract.json` and the `data.py` adapter. For the failure evidence and rationale, read `docs/事例_あかつきserver断片と失敗.md`. Invoked by judge-loop; fire grill-me for every fork.

Three modes across the lifecycle: diagnostic (developer), GT-creation (human), evaluation (steady-state, once gold/holdout exist). Never a finish line. The developer diagnostic + evaluation modes are ONE config-driven server (anti-sprawl is developer-side — S1); the human GT-creation surface MAY be a separate clean deliverable for handover (its own folder/package/server), unified to the rest not by one process but by the single contract source (S2) and single GT flow-back (S9). Whether GT-creation is a mode of the dev server or a split deliverable is a grill hook; the gates (S2/S3/S6/S9) hold either way.

Premise: real/human GT arrives LATE. The server exists so GT accrues incrementally — the developer saves correct/incorrect in diagnostic mode against the proposer (Claude) GT, and blind reviewers add independent GT — not so a finished GT can be loaded. Production judge is a local/open Gemma-class model via API (non-deterministic), shown only after commit (S3/S4).

## Gates vs choices
- GATES (S1–S12): always-on, non-negotiable. Never let the owner opt out.
- CHOICES: fact-checker surface form (a mode of the dev server vs a separate handover deliverable — S1/S2/S3 hold either way; lean split when external non-technical reviewers and clean handover matter), auth strength (default NONE — mode is a route; defer login/named reviewers until untrusted external blind reviewers arrive, do not pre-build a login screen), contract-file format, GT-tier schema shape, GT protocol per campaign, port/process layout, distribution medium. Never pre-decide — fire grill-me.

## Gates (enforce throughout)
- S1 One DEVELOPER diagnostic server, config-driven. The anti-sprawl lesson is developer-side: never a new program per verification angle (cross-item / keydiff / flutter / checklist / ... — that was the akatsuki sprawl). New angles are modes/lenses; domain is injected from config. SCOPE: this gate binds the developer surface. The human fact-checker GT-creation surface is a SEPARATE, clean deliverable (its own folder/package/server) and was singular by nature — do not fold it into the dev server merely to honor anti-sprawl; that mis-inherits a developer-side lesson into a context where it never held. What unifies the two surfaces is NOT one process but the single contract source (S2) and the single GT flow-back path (S9). Splitting them is fine — even better for handover hygiene and the firewall (S3) — provided S2/S9/S6 hold across the seam.
- S2 Single contract source. Axes, cardinality, vocabulary, field names, version stamps live in ONE file read by both the judge and the server. The server never hard-codes vocabulary; when the judgment changes the UI follows automatically. This is the invariant that survives an S1 split: a SEPARATE fact-checker deliverable must still read the SAME contract — its copy is GENERATED from the single source (version-stamped), never hand-edited. The most expensive akatsuki failure (F2) was exactly this seam: the separate human server re-encoded the contract and drifted (re-introduced an abolished term; read stale field names → blank human-review). The crime was contract DUPLICATION, not a second server.
- S3 Two roles, anchoring firewall. The firewall is RENDER-TIME (or by-absence when split), not auth: the GT-creation path never renders machine output. Developer/diagnostic mode sees everything and may save verdicts against the proposer GT — since it sees the machine, those are provenance=anchored (silver), never gold. GT-creation mode shows input + evidence only — no machine output (neither proposer nor production judge) until the human commits their own verdict + reason (provenance=blind → eligible for gold). Mode is chosen by route, NOT a login: a single developer runs the fast loop early; human blind reviewers arrive late. Do not pre-build a login screen / passwords / named reviewers — auth strength is a deferred grill hook, wired only when untrusted external reviewers need sequestration or attribution. STRONGEST firewall form = by ABSENCE: when the fact-checker surface is a separate deliverable (S1 split), the handover package simply contains no machine answers (S10 excludes the answer DB), so blind concealment is structural, not render-time or auth — a reviewer cannot reach machine output because it is not in their deployment; reveal/divergence is then computed developer-side when the GT flows back (S9). Render-time concealment is the within-one-server fallback when the surfaces are not split.
- S4 Reveal-after-commit drives the divergence queue. After commit, reveal proposer + production judge and surface human⇔proposer⇔judge disagreement. Divergence cases feed the next GT. Original blind verdict is append-only; revisions are new entries.
- S5 GT is append-only, disposable, cheap to regenerate. Never invest in careful GT.
- S6 GT maturity is tracked, versioned, never conflated. Tiers: bronze (model-generated seed) / silver (anchored ratify, screening-only) / gold (independent, blind-then-revealed) / red (landmines) / holdout (generalization check). Provenance per entry: blind/anchored/synthetic. Enforced in code: anchored ⇏ gold or holdout; red and holdout are separate stores; every gold entry carries criterion_version and is flagged stale → re-ratify or demote when the contract moves.
- S7 Review unit follows the judgment unit. Review unit (per-item / cross-item cluster / event) is config; it reconfigures when the unit moves in the spiral.
- S8 Reads are side-effect-free. Viewing never mutates state. State changes only on explicit write.
- S9 One ingestion path. Exactly one way data enters. No second manual-import path.
- S10 One verified distribution command, provenance always visible. The package excludes the aggregator's answer DB, others' submissions, caches, internal docs. The UI always shows live-vs-snapshot + timestamp. Manage ports/processes so no stale process serves old UI. Ship a preflight (a "doctor") that validates the receiving side with the REAL parser — environment + data STRUCTURE, go/no-go by name; an unverified handoff fails with a different symptom every time.
- S11 Evidence-confront, not agree-toggle. The human confronts the cited evidence; the verdict is captured against evidence pointers.
- S12 Once GT exists, evaluation is measurement, never a target. Never tune the judge to climb agreement with gold/holdout. Holdout is sequestered: hidden during development, surfaced only at milestones, every access logged.

## Walk
- W0 Read the contract (P1) and current unit (P0u) from judge-loop. If no single contract source exists, raise it as a judge-loop gap; do not hard-code around it.
- W1 Config schema: axes, cardinality + vocabulary per axis, field-name map, review-unit definition, evidence-pointer form, version stamps (unit/criterion/contract/judge). One file (S2). Grill: file format.
- W2 Two modes, chosen by route (no login by default — S3). Developer (`/diag`): 3-pane (left unit list / center input with all evidence highlighted + click-to-jump / right every judge with verdict + reason + evidence links) + aggregate (distribution + error clusters + cross-run flutter). During unit discovery it captures a round-indexed owner-reaction log — each reaction tagged unit-keying / criteria / noise + action — and computes SETTLE (K consecutive rounds with no unit-keying tag). This reaction log (spike/) is distinct from the GT store (verdicts) and the decisions ledger (ADR). GT-creation (`/review`): input + evidence only; capture verdict + reason against evidence (S11); reveal on commit (S4). Form factor: the developer surface may lean into IDE ergonomics (3-pane + keyboard next/prev + native diff for A/A·A/B·flutter + a Problems panel = divergence queue) — its mental model IS code review (navigate many units, evidence = go-to-definition); the GT-creation surface stays deliberately MINIMAL (anti-IDE — the firewall demands an information-poor view; an IDE's show-everything instinct fights blindness). Grill (deferred): auth strength — only when external blind reviewers need sequestration/attribution; default none. Real IDE integration (units as files in the developer's own editor) is a separate deferred option, not built before the unit settles.
- W3 GT store (S5/S6) + ingestion (S9). Append-only tiers + provenance; promotion rules in code (anchored ⇏ gold/holdout; red/holdout separate). One ingestion path. Grill: tier-schema shape; protocol per campaign (blind/ratify/synthetic).
- W4 Divergence queue (S4): developer-mode proposer-vs-judge disagreements feed the GT-creation priority queue.
- W5 Snapshot + distribution (S10): freeze a snapshot for shared review, live for latest, provenance on screen; one verified package command; verify the handoff end-to-end before declaring done.
- W6 Evaluation mode (S6 lifecycle + S12), once gold/holdout accumulate — a third mode, not a new server: regression eval vs frozen gold (split noise from real difference); holdout milestone check (access logged); GT-staleness queue (criterion_version lag → re-ratify); promotion path divergence-queue → human GT → silver → blind re-confirm → gold. Grill: eval metric; milestone cadence; staleness threshold.
