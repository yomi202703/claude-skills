Stand up the one review server for a no-ground-truth judgment loop. Domain-agnostic: vocabulary, axes, unit, and questions are injected from config, never hard-coded. Start from `template/` (runnable skeleton, stdlib only, gates enforced in store.py + server.py); adapt only `contract.json` and the `data.py` adapter. For the failure evidence behind every gate, read `docs/事例_あかつきserver断片と失敗.md`. Invoked by judge-loop; fire grill-me for every fork.

Three modes across the lifecycle: diagnostic (developer), GT-creation (human), evaluation (once gold/holdout exist) — never a finish line. The developer diagnostic + evaluation modes are one config-driven server (anti-sprawl is developer-side, S1); the human GT-creation surface may be a separate clean deliverable, unified to the rest by the single contract source (S2) and single GT flow-back (S9), not by one process. Premise: real/human GT arrives late — the server exists so GT accrues incrementally (developer saves anchored verdicts against the proposer GT; blind reviewers add independent GT), not so a finished GT can be loaded. The production judge is a local/open Gemma-class model, shown only after commit (S3/S4).

## Gates vs choices
- GATES (S1–S12): always-on, non-negotiable. Never let the owner opt out.
- CHOICES (never pre-decide — fire grill-me): GT-creation surface form (mode of the dev server vs separate handover deliverable; lean split when external non-technical reviewers matter); auth strength (default none, mode is a route — defer login/named reviewers until untrusted external reviewers arrive); contract-file format; GT-tier schema; per-campaign GT protocol; port/process layout; distribution medium.

## Gates (enforce throughout)
- S1 One developer diagnostic server, config-driven. Never a new program per verification angle (that was the akatsuki sprawl) — new angles are modes/lenses. Binds the developer surface only; the human GT-creation surface may be a separate clean deliverable. What unifies the two is the single contract (S2) + single GT flow-back (S9), not one process. Splitting is fine — better for handover hygiene and the firewall — if S2/S6/S9 hold across the seam.
- S2 Single contract source. Axes, cardinality, vocabulary, field names, version stamps live in one file read by both judge and server; the server never hard-codes vocabulary, so the UI follows when the judgment changes. A separate GT-creation deliverable generates its contract copy from that source (version-stamped), never hand-edits it. The costliest akatsuki failure (F2) was contract duplication, not a second server.
- S3 Two roles, anchoring firewall — render-time (or by-absence when split), not auth. GT-creation shows input + evidence only; no machine output until the human commits verdict + reason (provenance=blind → gold-eligible). Diagnostic mode sees everything; verdicts saved there are provenance=anchored (silver), never gold. Mode is a route, not a login — do not pre-build login/passwords/named reviewers; wire auth only when untrusted external reviewers need sequestration/attribution. Strongest firewall = by absence: a split handover package simply omits the answer DB (S10), so concealment is structural and reveal/divergence is computed developer-side when GT flows back (S9).
- S4 Reveal-after-commit drives the divergence queue. After commit, reveal proposer + production judge; human⇔proposer⇔judge disagreement feeds the next GT. The blind verdict is append-only; revisions are new entries.
- S5 GT is append-only, disposable, cheap to regenerate. Never invest in careful GT.
- S6 GT maturity tracked, versioned, never conflated. Tiers: bronze (model seed) / silver (anchored ratify, screening-only) / gold (independent, blind-then-revealed) / red (landmines) / holdout (generalization). Enforced in code: anchored ⇏ gold or holdout; red and holdout are separate stores; every gold carries criterion_version and is flagged stale → re-ratify or demote when the contract moves.
- S7 Review unit follows the judgment unit (per-item / cross-item cluster / event) — config; reconfigured when the unit moves in the spiral.
- S8 Reads are side-effect-free. State changes only on explicit write.
- S9 One ingestion path. No second manual-import path.
- S10 One verified distribution command, provenance always on screen. The package excludes the answer DB, others' submissions, caches, internal docs; the UI shows live-vs-snapshot + timestamp; no stale process serves old UI. Ship a "doctor" preflight that validates the receiving side with the real parser (environment + data structure, go/no-go by name) — an unverified handoff fails with a different symptom every time.
- S11 Evidence-confront, not agree-toggle. The human confronts the cited evidence; the verdict is captured against evidence pointers.
- S12 Once GT exists, evaluation is measurement, never a target. Never tune the judge to climb agreement with gold/holdout. Holdout is sequestered: hidden during development, surfaced only at milestones, every access logged.

## Walk
- W0 Read the contract (P1) and current unit (P0u) from judge-loop. No single contract source → raise it as a judge-loop gap; do not hard-code around it.
- W1 Config schema in one file (S2): axes, cardinality + vocabulary per axis, field-name map, review-unit definition, evidence-pointer form, version stamps (unit/criterion/contract/judge). Grill: file format.
- W2 Two modes by route (no login — S3). Developer `/diag`: 3-pane (unit list / input with evidence highlighted + click-to-jump / every judge with verdict + reason + evidence) + aggregate divergence queue; may lean into IDE ergonomics (keyboard next/prev, native diff for A/A·A/B·flutter, a Problems panel) — its mental model is code review. GT-creation `/review`: input + evidence only, capture verdict + reason against evidence (S11), reveal on commit (S4) — deliberately minimal (anti-IDE; the firewall demands an information-poor view). During unit discovery the developer surface also captures a round-indexed owner-reaction log (spike/, distinct from the GT store and the decisions ledger) and computes SETTLE (K rounds with no unit-keying tag). Deferred grills: auth strength; real IDE integration (units as editor files) — not before the unit settles.
- W3 GT store (S5/S6) + ingestion (S9): append-only tiers + provenance, promotion rules in code (anchored ⇏ gold/holdout; red/holdout separate), one ingestion path. Grill: tier-schema; per-campaign protocol.
- W4 Divergence queue (S4): developer-mode proposer-vs-judge disagreements feed the GT-creation priority queue.
- W5 Snapshot + distribution (S10): snapshot for shared review, live for latest, provenance on screen, one verified package command; verify the handoff end-to-end before declaring done.
- W6 Evaluation mode (S6 + S12), once gold/holdout accumulate — a third mode, not a new server: regression vs frozen gold (split noise from real difference); holdout milestone check (logged); staleness queue (criterion_version lag → re-ratify); promotion divergence-queue → human GT → silver → blind re-confirm → gold. Grill: metric; cadence; staleness threshold.
- W7 Rubric-frontier lens (an owner-audit view, generated — not a new program: S1; not a new gate). Render the criteria evolution from ONE source: decisions + the P4 atomic criteria ledger (S2 — generated, never hand-edited). Shows current active criteria (frontier) vs superseded history; each revision tagged with its one breaker case; merges/splits/supersessions allowed (a revision history, not a causal graph). Owner-language only (internal id/schema kept off-screen, mapping in a separate layer). Read-only (S8). Developer/owner surface only — never the blind `/review` surface (showing the evolving rubric there would anchor the reviewer, S3). Surfaces the criterion_version lineage and staleness that W6 re-ratifies against. Deferred until the host actually needs to audit criteria evolution; this is documentation of the lens, not a template build.

## Composition
- Every fork → grill-me.
- Source preprocessing for what the server displays → xlsx-router, pdf-to-md.
- Invoked by judge-loop: P0u / W6 are developer modes of the one server; P2 GT-creation is a mode of it or a separate handover deliverable (S1/S2/S9 hold either way); the P4 rubric-frontier lens (W7) is a generated owner-audit view, a lens not a new program (S1/S2/S8).
- Rendering discipline for the judgment/score surfaces → html-deck (raw numbers + threshold coloring not verdict-label columns, funnel spine, priority ordering, dual-encoding, owner-language). Applies layered, gated by S3:
  - /diag (divergence queue, per-unit judge panel) and evaluation mode (regression/agreement/staleness) → full html-deck; the divergence queue is a judgment table, evaluation is a score table.
  - W7 rubric-frontier → html-deck principle "相手の言葉で書く"/internal-representation removal + craft.
  - Blind /review (S3) → only html-deck's craft/legibility/owner-language layer. Its judgment-rendering layer (coloring verdicts, a funnel of machine results) is firewalled out pre-commit — applying it would breach S3. After reveal, the divergence view takes full html-deck.
  - Synergy: html-deck's "no internal representation" is operationalized by S2 (server reads vocabulary from the contract, never hard-codes) — S2 makes that principle structural, not aspirational.
  - Non-conflict: tier pills (gold/silver/blind) are provenance metadata with no underlying number, so they are not the verdict-chip html-deck forbids (it forbids chips that hide a raw judgment number).
