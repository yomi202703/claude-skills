## 1. プロジェクト全体像

- 目的: 口座番号(6桁・主キー)に、接触履歴・通話音声(文字起こしmd)・取引データを突合し、証券営業の規制遵守等を判定。判定エンジン = Gemma(ローカルLLM・gemma-4-31B-it)＋決定論ロジック。
- 3層リポジトリ:
  - `pipeline_13_音声判定統合/` — 判定ロジックの開発・検証(SQLite)。新規ロジックはここ。
  - `deploy/` — 本番 ETL＋Web UI(MySQL/RDS＋AWS Lambda)。参照側出力を輸入して配信。新規ロジックは書かない。
  - `_data/` — 機密(gitignore)。
  - `factcheck/` — AI判定を人手で確認するレビューツール(独立・read-only)。
- 起動最短: `pip install -e .` → `source .env`(AKATSUKI_LLM_*) → `common/doctor.py`(受領側プリフライト) → `common/run_account.py <口座>` → `common/viewer.py`(:8012)。
- 配布は repo＋_data 一本(2026-06-20 に zip/UI取込を全廃・全口座~34分で誤差ゆえ判定済み配布不要に)。

## 2. 判定システム（成果物の4本＋付随）

判定単位(unit)が各モジュールの肝。

- 心情(shinjo) = 感情分析。単位 = (通話×銘柄)・売買検討のみ(注文管理除外)。出力 = polarity(ポジ/中立/ネガ)＋strength(強/弱)をGemmaが出し、emotion 1-5 はコードで決定論変換(v7)。actionable フラグ = level1(不安・懸念)のみ。1通話1コール一括。最終分布(全29口座1056銘柄) 概ね 1:17-32/2/3:過半/4/5・中立を頂点に山型。
- 主体性(shutaisei) = はいはい取引。単位 = (通話×銘柄)・売買検討∧勧誘ありのみ。営業主導=顧客が承諾語だけで他の関与ゼロ、に厳格化(過大判定を是正・営業主導率42%→~10%)。分布 営業主導37/協働53/顧客主導3(93銘柄)。
- 認知(ninchi)。70歳以上のみ。最も単位が暴れた(3章)。最終 = 固執(within-call の reassert>=2)∪認識失敗(銘柄を gap≤7日でまたぐ密な失念)。口座総合は2値(要確認/懸念なし)=「懸念あり」は越権ゆえ廃止。最終分布 要確認5(310048/539661/517626/772984/529405)/懸念なし17。
- 規制遵守(kiseishuuji) = チェックリスト。単位 = (義務×取引イベント)・マルチチャネル証拠。原典PDF突合で商品別スクリーンに原子分割(投信TS/外国株GK/外債GB/IPO IP/仕組債SB/レバLV/信用SY)。説明義務 = 抽出＋二値＋判定不能(省略可否が録音外で確定不可)。禁止発言 = 再設計2スクリーン(確実断定 ScreenB＝C-032価格/C-036信用＋金銭的便宜 ScreenA＝C-034/045/075/127)。
- 乗換(norikae) = SWAP。trades 決定論でイベント検知→紐づく通話を Gemma 判定。kisei から分離(単位がイベントで通話単位 kisei と噛み合わなかった)。
- call_facts = 共有事実層。通話1本Gemma1コールで 銘柄/勧誘区分(solicit)/phase/記録要否 を抽出。全モジュールの共通入力。
- onsei_kiroku = 音声記録整合。現在判定対象外(記録漏れは企業の記録不備の言い換えと判断)。コード保全。

## 3. 認知モジュールの変遷（最も学びが多い・2026-06-14〜06-21）

判定単位が一週間で十数回作り替わった記録。これが「正しい単位の発見が最大の時間」の実例。

- 起点: 旧設計は「記憶の再確認カウント」を採点していたが、これが文字起こしから最も不安定(precision 28-55%)と定量判明。
- 単位の遍歴: severity合計(別種を足す)→ fact_type別 → 通話内反復(reps) → error_idx クラスタ → reassert(訂正後の再断定) → recur(認識失敗) → (銘柄,aspect) → 銘柄のみ → gap密度(≤7日)。
- 決定的なオーナー裁定:
  - 「なんで別種を合計しているんだ」→ severity合計を廃止(単位の誤り)。
  - 「期間が短いほど怪しい」→ gap≤7日の密な失念だけカウント(532856 中村超硬の別事実束ねFPが消える)。
  - 「全然忘れている対象が違う」→ aspect/銘柄で同一事実を切る。
  - 「会話だけで AI が認知症を断定するのは越権・医者でも会話だけでは断定不能」→ 口座総合を2値化(懸念あり廃止)。
    ※これはこのドメインの現実(会話だけでは確定不能・高 stake)を見た 一事例の選択 であってゲートでない。
      断定禁止/2値/隔離 を他ドメインへ継承しない(G4 はこれを非ゲート化済・横断方法論 doc 参照)。
- 再現性の授業料: 06-19v「完全再現で安定を実証」を 06-19y で K=5 実測により訂正(完全再現は n=1 の運)。境界口座は run で判定が入れ替わる。
- 軸の妥当性 vs 検出性: orientation 軸を「締めても precision 25%・磨くこと自体が overfit」と撃退。「再現性が高い軸だけ残すのは街灯バイアス」というオーナー指摘。
- 銘柄名寄せ: 表記揺れ(NEXTERA romaji↔カナ)で genuine グループが分散→銘柄マスタ(canon resolver)＋Wikidata カナ読み表で systematic 化。
- GT: 認知_オーナー判定GT.md(オーナー裁定・出典idx付き)。旧 per-call labels.json は基準が動いて2回廃棄。最後の8口座は Claude subagent が独立精読→オーナー裁定(GTは判定プロンプトに入れない=eval汚染防止)。

## 5. 横断的な確定事実

- Gemma 非決定: temp=0 でも約9%の揺れ床(規制 C-032 で実測)。境界ケースは run で verdict が反転。多数決(votes)は境界FPを消さず入れ替えるだけ(が faint な本物の recall 安定には効く)。
- viewer 群(俯瞰計器): 本体 viewer.py(:8012・3ペイン)＋ eval 用 cog_xcall_viewer(生全文＋ハイライト)/cog_flutter_viewer(K列で判定ブレ)/ab_diff_viewer/run3x_viewer/ab_review_server(before/after)。単位vs基準を可読化する診断計器。
- ドキュメント統治(グローバル方針): TODO(未来の queue・2状態 Active/Deferred)・STATUS(現況スナップショット・rewrite)・decisions(append-only ADR・トピック別)・archive(凍結)。「やらない」は decisions に1エントリ・TODO からは消す。
- データモデル: 口座別 `outputs/<口座>/<口座>.db`(accounts/trades/holdings/audio_files/発話/contact_full/call_facts 等)。接触履歴 = `_data/接触履歴/{口座}.csv`(cp932・54列)。音声 = `_data/音声/{口座}/*.md`(全1042件/29口座)。
- 銘柄 canon: firm_union(49顧客の取引)＋TSEマスタ(data_j.xls)＋curated alias(aliases.json)＋Wikidata 由来カナ読み(foreign_auto.json・82%自動取得)。証券コードで名寄せ。
- 配布/再現性: doctor.py が受領側プリフライト(依存/.env/_data形を本物パーサで検証)。契約は _data 一本。
- run_account の構成: preprocess→bootstrap_db→route→broadcast→各判定。プロンプト(*.md)とロジック(.py)は viewer で importlib.reload=編集即反映。

## 6. factcheck ツール

- 目的: pipe13 の AI 判定を人の目で確認し、判断基準を複数 reviewer で固める。判断基準そのものを集約してルール化の原資料に。
- 設計: 標準ライブラリのみ(http.server＋pyyaml)・SQLite 1ファイル(factcheck.db)・追記オンリー・CSV を集約役が inbox でマージ(RDS不要)。:8020。
- outputs/ を read-only で読む(AI判定・通話記録は読むだけ・既存パイプラインに触らない)。回答に ai_verdict_json(焼き付け)＋ai_produced_at＋created_at。
- 設問: Q0判定要否・Q1主体性・Q2心情(銘柄ごと)・Q3認知・Q8禁止文言(通話)・Q7特記・まとめQ5認知総合。賛成/反対/保留。
- 重要: これは「人間が正解(GT)を作る」サーバー。あかつきでは開発者の「検査IDE」(viewer 群)と物理的に別プログラムだった(史実)。ただし標準(review-server)はこれを畳むべき sprawl とみなし、診断/GT作成/評価を1サーバーの別モードへ統合する=史実は「分離」、標準は「統合(commit後開示の防火壁つき)」。混同しないこと。

## 7. 主要な教訓（あかつき具体版・方法論の元ネタ）

1. 単位を継承せず目標から導く。legacy per-contact 枠を1ヶ月再現したのが最大の浪費。norikae のイベント錨型を全取引に一般化して解決。
2. 正解は事前に無く、物差し(GT)も動く。labels.json を2回廃棄。検証は label-free A/B＋目視＋K再現に一本化。
3. 基準はオーナーが出力を見て1例ずつ構成する(動く的)。台帳のほぼ全エントリが「オーナー指摘」起点。
4. Gemma は基準を誤読(断定=押しの強さ)＋非決定。語をプロンプトに置くと誘導される(「不安懸念」→数値→polarity+strength)。
5. 境界は人間でも引けない(言い切り81%○・専門家が真剣に見ても線引き不能)。要確認(人手枠)で吸収。
6. masking(束ね)は原子分割で割る(原典突合)。
7. load 敏感: carve-out を足すと推論型ケースが静かに落ちる(562420=総制約量)。句を足さず別スクリーンに分離。
8. 基準の問題に見えて証拠の問題(銘柄 anaphora・表記揺れ・信用は tx_type)。
9. 妥当性≠検出性(街灯バイアス)。再現性で軸を選ぶと overfit。
10. 過学習禁止(同一サンプル追いチューニング不可・holdout 検証)。主体性ホールドアウト適合率80%。
