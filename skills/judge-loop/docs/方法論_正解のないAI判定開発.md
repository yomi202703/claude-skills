# 方法論 — 正解のないAI判定システム開発

正解(ground truth)が事前に存在せず、判断基準そのものをオーナーから1例ずつ構成していくタイプのAI判定システム開発を、再現性をもって速く・安全に回すための方法論。最終的には「1つの skill を起動すると最適なループが立ち上がる」ことを目指す。

対象は金融に限らない——正解データが無い判定 全般のドメイン非依存な標準を狙う。定義4条件: ①事前GTなし(正解は人間が構成するまで存在しない) ②基準は暗黙知でオーナーから case-by-case ③専門家でも割れる/境界が曖昧 ④LLMで大規模にやる。例: コンテンツモデレーション/ポリシー適合・臨床トリアージ/懸念検知・候補者/小論文/業績評価・品質/リスク/感情スコア・「これは適切か/問題か/違反か」系 全般。あかつき(感情/主体性/認知/チェックリスト)はこの class の一事例にすぎない=標準が事例を継承して狭まる(過ち1/G1)のを自ら避ける。狙い=prompt-eng / context-eng と同列の一般実践=標準化。

本書は確立途中の設計ドキュメント(working design)。確定した判断と、次に詰める論点を分けて残す。

---

## 1. 目的（観測可能な変化）

生むのはコードでもGTでもなく、回り道の短縮。新しい曖昧判定タスクに着手してから「根拠つき・確信度つき・人手審級つきの安定したスクリーニング出力」が出るまでの時間を、あかつきで要した数ヶ月＋一週間の試行錯誤から、skill 起動で大幅短縮する。具体的には、あかつき認知判定で踏んだ回り道(軸の overfit・再現性の測り直し・aspect 廃止・2値化への到達)を次のドメインで踏ませない。

---

## 2. 診断 — なぜ判断基準がぶれ、詰めるのに時間がかかったか

開発史から抽出した原因。大半は no-GT 判定開発に構造的に内在し、領域非依存(=方法論は汎用化できる)。

### 台帳期(pipe13・6/14〜6/21)の8原因
1. 判定単位を「選ばず継承して」作り込んだ(最深)。正しい単位の発見が最大の時間。
2. 正解がなく、物差し(GT)自体が動いた。per-call 人間GTは2回廃棄。
3. 基準は仕様でなく、オーナーから1例ずつ引き出すもの(viewer を見て反応)。動く的。
4. モデル(Gemma)が基準を誤読(断定=押しの強さ)＋約9%の非決定。幻のリグレッションを追った。
5. 境界は本質的に引けない。人間でも引けない線を引こうとし続けた → 最後は要確認(人手枠)で割り切り。
6. masking(束ね)。束ねた基準は「部分的にある」に化ける → 原典突合で原子分割。
7. モデルの load 敏感性。carve-out を足すたび別の本物が静かに落ちる。
8. 基準の問題に見えて実は証拠の問題(銘柄 anaphora・表記揺れ・信用は tx_type)。

### 台帳以前(pipe1〜12・5月)の4過ち
1. 判定単位を legacy から継承し1ヶ月「完全再現」した(最大の浪費)。
2. legacy の道具立て(数値確信度閾値・多軸・LLM下請け)を丸ごと持ち込み、後で全部捨てた。
3. WHY を残す append-only 台帳が無く、記憶が生ログ＋17回書き換えの STATUS だけ → 蒸し返し。
4. 早すぎる共通化(pipe9 解体)。軸が固まる前に横断インフラを作った。

## 3. 確定した設計判断（このセッションのADR）

- D1: reuse の単位 = プロセス(skill)主体。方法論は「セッションを駆動する規律＋ゲート」として持ち運ぶ。コードは雛形だけ出力。単位発見とオーナー例示は人手のまま、回り道を踏ませない。参照実装(あかつき)をテンプレ複写するのは過ち1の再演ゆえ不採用。
- D2: 素性は1軸でなく直交2軸。
  - 軸1 アーキテクチャ(単位・証拠範囲・出力契約): 目的から導けることがある(規制=導けた / 認知=導けず)。GT可否と無関係。
  - 軸2 基準(何を違反/懸念/問題とするか): no-GT では本質的に常に創発(オーナーが case-by-case で構成)。
  - 重要: 「専門家がルールを言えるか」と「GTを作れるか」は別物。判定の中身で定義型だったものは今回ゼロ。
    定義型が現れたのは規制のアーキ層だけ。
- D3: 単位は定義型/創発型で出し分け。定義型は導出(継承拒否)、創発型は螺旋へ即移行。
  迷ったら創発型に倒す(誤分類の非対称: 創発を定義型と誤れば数週間溶かす / 逆は安いスパイク1回の無駄)。
- D4: P0.0「源泉の取り込みと棚卸し」を P0 の前段として新設(下記4章)。
- D5(メタ原則 M1): 方法論は「ゲート」と「選択」に割れる。
  - ゲート(always-on・非交渉) = G1-G9。過ちから導いた不変条件。選ばせない。ここが梃子(leverage)。
  - 選択(fork・ケースバイケース) = 出力の役割・単位の粒度・確信度の出し方 等。
    AIが「選択肢＋各々のトレードオフ(あかつき代償)＋根拠ある推奨」を出し、決定はオーナー。AIの提案は1入力。
  - 注意: 全決定を「ケースバイケース」にすると skill は「全部オーナーに訊く」になり梃子ゼロ。
    方法論の価値は決めることでなく、選択肢を漏れなく枠づけし推奨を添えること。この線引きが背骨。
  - 選択は「いま答えを埋める」のでなく、skill が該当ステップで発火させる grill hook として実装する
    (AIが選択肢＋トレードオフ＋推奨を提示・オーナーが use-time に決定)。方法論は答えでなく「grill する場所」を持つ。
    例: GT 生成機構は P2 の use-time grill hook で決め、設計時には焼き込まない。

## 3.5 Scaffold — 最適ディレクトリ構造（day1・P0.0 の前段）

あかつき実構造から「効いた型」を抽出し、「痛かった所」(GT 二重在住・契約散在)を構造で潰した canonical tree。
domain-agnostic。判定固有部は judge-loop が置き、CLAUDE.md と 成果物/(統治)は claude-md へ委譲する。

```
<repo>/
  CLAUDE.md                    ← claude-md(repo 層を著述)
  .env.example                 ← commit。Gemma API 接続テンプレ(LLM_BASE_URL/API_KEY/MODEL)
  .env                         ← gitignore。実値。判定ランタイムが source して読む
  .gitignore                   ← claude-md 管理。.env / _data/ / outputs/ / snapshot / 回答DB
  _data/                       ← 生源泉・機密・gitignore
    raw/{xlsx,pdf,transcripts}/
    processed/                 ← xlsx-router / pdf-to-md の出力先
  access/                      ← Pre-P0.0 統一アクセス層(MCP)
  contract/                    ← P1 単一契約ソース(S2): 軸/語彙/単位/version
  factlayer/                   ← 中立な共有ファクト抽出(1回抽出→全 judge が消費・ロール中立=違反探しでない・あかつき call_facts)
  <axis-1>/ … <axis-N>/        ← 判定軸ごと1モジュール(factlayer を入力に)
    judge.py                   ← 決定論 + Gemma 呼び(local API)
    prompts/                   ← gemma-prompt の出力
    tests/
  gt/                          ← GT 倉庫を1ヶ所に(散在の修正)
    <axis>/{bronze,silver,gold,red,holdout}/   + provenance / criterion_version
    divergence_queue.*         ← S4 食い違いキュー
  spike/                       ← 単位スパイク反応ログ(round 付き・診断タグ unit/criteria/noise・SETTLE 計算の基質)
                                  review-server 診断モードが捕捉。GT(verdict)とも台帳(ADR)とも別物
  outputs/<unit>/              ← 判定結果(judge と review_server が読む)
  review_server/               ← review-server template の実体(contract 参照 / data.py / snapshot / inbox)
  eval/                        ← P3 ハーネス(K-run/flutter/分布diff・測定計器であって本番経路でない)
    run_ledger.*               ← run 結果の恒久記録(version stamp judge/prompt/contract/criterion・drift%・反転ケース・noise/regression/improvement)
    runs/                      ← raw run(剪定: 直近N＋ベースライン＋境界のみ・剪定内容は ledger に1行記録)
  成果物/                      ← claude-md が雛形生成・global 統治
    STATUS.md  TODO.md
    decisions/{<axis>.md, 横断.md, README.md}
    archive/
```

肝:
- GT は `gt/<axis>/<tier>/` に一元化＝あかつきの二重在住(module/eval と 成果物/ の両在)を構造で禁止。outputs(機械判定)と gt(人/Claude 正解)を分離。
- contract/ を単一ソース化＝F2/S2。judge も review_server も同じ契約を読む(陳腐語の再導入を防ぐ)。
- xlsx-router/pdf-to-md は `_data/processed/` へ吐く規約に固定＝取り込み合成が噛む。
- 所有: 判定固有(source/軸/contract/gt/outputs/review_server/eval)=judge-loop / 記憶層(CLAUDE.md・.gitignore・成果物/)=claude-md。
- Gemma API 接続は .env(gitignore)＋.env.example(commit)＝判定ランタイムの関心。gemma-prompt(プロンプト設計のみ)には入れない=スコープ分離。

## 4. P0.0 — 源泉の取り込みと棚卸し

判定を組む前に、もらった全源泉を網羅的に取り込み「これにはこれがある」の地図を作る。
原因8(証拠問題)と規制の channel-error(義務を誤チャネルに載せ×量産)を構造的に潰し、
データの grain から単位候補を炙り出す。

### アクセス層は3モード（材料×役割で使い分け）
- 構造化データ(trades/accounts/接触履歴) → SQL query。判定の証拠供給。
- 原典マニュアル(PDF) → ページ指定の直読(TOC 辿り・原典突合)。基準導出。RAG は原典が巨大なときのみ。
- 文字起こしコーパス → 意味検索(RAG)。発見・exemplar・反例・候補基準の横断発火確認。
  ※判定時は通話まるごと全読みに切替。RAG が要るのは発見・螺旋フェーズ(P0.0/P0u)であり判定(P3)ではない。
- 統一アクセス層 = MCP server。上記3つを判定/grill エージェントが同じ作法で叩く口。
  「RAG基盤」ではなく、文字起こし探索モジュールを含む統一アクセス口、が正確。

### 出力 = 型つき源泉地図
各源泉に役割タグを振り、方法論のどこへ流れるかを決める。
- evidence(判定が見る・チャネル別) / authority(基準の出どころ・原典) /
  exemplar(サンプル=仮GT種・期待挙動) / meta(口座属性=決定論ゲート入力)
- さらに「原典の基準被覆率」を測る: 原典が在ることと、原典が自分の基準を定義していることは別。
  規制=原典が基準の大半を定義 / 認知=コンプラ原典は認知基準を一行も定義せず(全面螺旋)。
  PDF の存在=基準導出可能、と早合点しない。被覆分は導出、非被覆分は螺旋。

## 5. 方法論の構造（骨格）

### 縦軸：フェーズ
- P0.0 源泉の取り込みと棚卸し（4章・確定）
- P0u 単位発見(螺旋・最重要): 単位・証拠範囲・出力契約を目的から導き継承を剥がす(定義型) / no-GT 基準は常に創発と置き仮GT・再現ハーネス・反応ログの足場を立てる(創発型)。データの grain が単位候補。実史で P0a/P0b は分離せず1つの螺旋=単位スパイクで収束。D2/D3: 定義型→導出(継承拒否)・創発型→即螺旋・迷ったら創発。
- P1 出力契約（**確定**・各項目＝原理(ゲート)＋形式(選択) に分解）:
  - 根拠: 証拠を指す[ゲート] / ポインタ形式(idx・行ID・ルールID)[選択]
  - 確信度: 生の数値自己確信を出させない・数値はカテゴリor構造からコード導出[ゲート] /
    有無と形式{なし / 構造導出ティア(認知型) / モデルのカテゴリ段階→決定論変換(心情型・現役)}[選択]
  - 隔離: 形式(3層参考=認知 / 単一閾値=心情 level1のみ)[選択・ドメインの現実で決定]
  - 枠づけ: 診断 vs 仮説・断定の可否・人手 routing[選択・ドメインの現実で決定]
    ※ゲートに残すのは「証拠を引いて検証可能」と「証拠/モデルが支える範囲しか主張しない(過剰主張禁止)」だけ(G4)。
      断定禁止/2値/隔離 は あかつき認知で効いた一事例の選択であってゲートでない=継承しない。
  - 安定: 契約を早期固定(P0u SETTLE 後)・変更稀(変えると累積GT/labelが無効化=原因2)[保護する錨=hard gate でなく phase 規律。変える時は GT 無効化の代償を承知で]
  - cardinality: actionable 旗の段数{2値 / N段 / 連続+閾値}[選択・standing推奨=2値スクリーニング]
- P2 仮GT＋検定（**確定・優先順位反転**・1サーバー2モード）: 成果物は「careful なGT」でなく「生成→雑GT→人間確認後段 まで end-to-end で回る環境」。実体は review-server の1サーバー(診断モード＋GT作成モード)であって別ビルドでない。
  - GT は使い捨ての種(precious でない・単位/基準が動けば気軽に捨てる)。labels.json で高くついたのは GT を作ったことでなく投資したこと。
  - アンカリング防火壁(review-server S3/S4): GT作成モードは人間が自分の verdict＋理由を commit するまで機械出力(提案モデルも本番judgeも)を一切見せない・commit 後に開示して食い違いキューを駆動。機械を先に見せると賛成9割化(アンカリング=eval汚染とは別の失敗)。
  - GT protocol は成熟度別(review-server S6): gold/holdout=ブランクスレート＋commit後開示(independent) / silver/triage=提案モデルの reason を見せ高速 ratify(anchored タグ・gold/holdout 昇格禁止) / bronze=モデル生成の種。reason は証拠(idx)接地・人間は reason vs 証拠 で確認(モデルは plausible に作話)。
  - 検定は up-front ゲートでなく P2 の連続 back-stage(人手審級)。GT は使われながら良くなる。「単位安定までGT作らない」は緩み「雑に作り気軽に捨てる」へ。
  - 生き残るゲート: GTは判定プロンプトに入れない(eval汚染) / judge を 提案モデル/GTに合わせ込まない(クローン=G3/G9・GTは測定用で reward でない) / anchored⇏gold/holdout・red≠holdout / 権威はオーナー。
  - どの protocol を使うか・GT の生成機構(広さ/単一・panel/ケースの集め方)は焼き込まず、use-time の grill hook で決める(今は詰めない)。
- P3 judge＋再現ハーネス（**確定**）:
  - ゲート: 再現ハーネス(K回独立ラン)を day1 配線(temp=0でも〜9%揺れ=規制C-032) / 幻のリグレッション防止(変化はまずノイズか実差か切り分け=認知06-19y) /
    ループを judge で閉じない・judge は reward でない(score を上げにいかない) / 揺れは安全側(本物の消失を許さない・境界は人手枠) /
    出力は P1 契約・生数値自己確信なし(G2) / judge と GT-builder(Claude)は独立(クローン回避)。
  - 効かないこと(授業料): 多数決(votes)は境界FPを消さず入れ替えるだけ(認知06-18) / ただし votes は faint な本物の recall 安定には効く(境界精度≠recall=道具が違う) /
    再現ハーネスは測定・診断の計器で production 経路でない(認知 production=votes1・俯瞰計器の flutter と共有)。
  - grill hook: production judge のモデル選定(コスト/速度/精度) / K と votes の有無 / 安全側の向き(ドメイン依存・最重要・焼き込まない=認知は見逃し>誤フラグ)。
  - run-ledger(記録の第5型・恒久): 全 raw run の保持は嵩んで限界=2層に割る。run-ledger=1 run-set 1エントリの恒久・追記オンリー{時刻・version stamp(judge/prompt/contract/criterion)・K・テストした変更・結果(noise/regression/improvement)・drift%・反転ケース＋方向・raw 参照}。raw runs=直近N＋現ベースライン＋人手キュー中の境界のみ保持・剪定し剪定内容を ledger に1行(沈黙の切り捨て禁止)。住所 eval/(run_ledger 恒久＋runs/ 剪定)。
    意義: drift を どの版変更に帰属させるか は GT 側 version(S6)＋run 側 version(ここ)の両方で初めて閉じる=幻のリグレッション殺し(G3/G9)の実体。decisions(ADR)/gt(verdict)/spike(単位反応)/outputs(機械判定) とは別の第5記録。
    how-to: ハーネスは judge-loop 自前の唯一コード。設計・run-ledger format は本節が骨子・実 template/code は deferred(6章)。
- P4 基準を詰めるループ（**確定**）:
  - ゲート: carve-out を足すたび full resweep＋俯瞰(G5・load敏感=規制562420) /
    基準はオーナー反応から引き出す(原因3) / 同一サンプルで追いチューニングしない・holdout/新規で汎化(G9) /
    却下理由を台帳に残す(G6) / 基準を疑う前にデータを疑う(G8=証拠経路＋原典被覆) / 引けない境界は引こうとせず要確認で吸収(原因5)。
    ※ masking 回避の原子分割は基準を割ってから締める手順だが、ゲートでなく gemma-prompt/P4 の技法(旧 G5 は原典被覆不足の症状=G8 へ統合)。
  - 効かないこと→別の手: 締めが本物を壊すとき(load敏感)、句を足すのでなく最小制約の別スクリーンに分離(規制ScreenA=金銭的便宜の分離)。
  - grill hook: precision vs recall の向き(ポリシー・オーナー裁定=認知recall重視/主体性tie-break再現率) / carve-out を足すか撤退するか(load敏感ゆえ毎回判断)。
- P2(人手審級＝GT作成モード・連続 back-stage): 人間が正解(GT)を作る場所=追記オンリー・集約。診断モード(俯瞰計器)とは別 surface でなく、review-server の同一サーバーの別モード=
  GT作成モードは答えを作る / 診断モードは全部を見て診断・比較する。最後の駅でなく(旧 P5「最後の駅」観は却下=P2 に統合)、ループを初日から end-to-end で閉じるため早く立てる。自動収束させない。
  診断モードで 提案GT⇔本番judge が食い違ったケースが、GT作成モードへ送る最優先キュー(commit 後開示で生成=S4)。
  GT 確立後は同一サーバーが評価モード(steady-state・review-server W6)を獲得=回帰評価/holdout マイルストーン/GT鮮度。終端でなく螺旋の cadence が落ちるだけ。反 overfit ゲート(S12=holdout を開発者から隔離)は GT 出現前に焼く。

### 横軸：不変ゲート（全フェーズに張る・継承拒否の番人）
ゲートは帰納で正当化される=複数ドメインの具体で成り立つもの。1ドメインで効いた規則(例: あかつき認知の断定禁止/2値)は次では選択であってゲートでない。各ドメインの現実を見てから当てる(理論先行で現実を見失わない)。
- G1 単位を継承しない（過ち1）
- G2 決定論ファーストのアーキ・LLMは意味の残差のみ・確信度はコード導出(モデル自己申告禁止)。プロンプト技法は gemma-prompt（過ち2・原因8）
- G3 本番 judge は報酬でない・自身のスコア/Claude/LLM評価器のクローンにしない・holdout で測る(測定であって target でない)・報酬はオーナー反応（原因2・4）
- G4 主張を現実に合わせる=証拠/モデルが支える範囲しか主張せず証拠を引いて検証可能にする。断定/2値/隔離はドメインの現実で決める選択（原因5・ただし断定禁止自体は認知一事例ゆえ非ゲート化）
- G5 基準を変えたら(特に carve-out)全件 resweep＋俯瞰で他ケースの巻き添えを検出（原因7・load 敏感のメカニズムは gemma-prompt 前提・救済は P4 で別スクリーン分離）
- G6 grill hook の却下オプションとオーナー裁定を理由つきで台帳へ流し fork を蒸し返さない（過ち3・台帳の構造自体は claude-md/global=本ゲートは choice 機構の出力を流す義務）
- G7 インフラはカレンダーでなく単位依存度で時を決める。単位非依存の基盤(scaffold・取り込み/データ理解・アクセス層)は単位が固まる前に作ってよい=単位が動いても形が動かず出戻りなし・P0.0 が早期に要求。単位依存のインフラ(delivery・作り込み serving・carve-out)は P0u SETTLE まで遅らせる。例外=診断計器: 単位依存だが早く作る(分布を見ないと単位を発見できない)→config 駆動で単位変更を再設定で吸収(単一 review server S1・viewer 乱立でない)。迷ったら「形は単位で決まるか」を問い・判別不能なら pre-SETTLE とみなし計器以外を遅らせる（過ち4=早すぎる共通化と取り込み早期化を非矛盾化）
- G8 基準を疑う前にデータを疑う=誤検知は証拠経路・抜け漏れ/masking は原典被覆(RAG/系統突合)（原因8＋旧 masking 原因6=実は被覆不足。原子分割は gemma-prompt/P4 の技法でゲートでない）
- G9 過学習禁止・validity≠detectability=再現性/測りやすさで軸を選ぶな(街灯バイアス)・holdout/新規オーナー反応で汎化確認・検証標本で追いチューニングしない（主体性 holdout・認知 orientation 撃退・実装は review-server S12）

### 構造の含意：単位↔基準の螺旋の扱い（**確定**）
P0〜P4 は実史では分離せず螺旋だった。解＝「単位スパイクで単位を先に収束→安定後に基準とGTへ投資」。

- 固定する錨(早期決定・滅多に動かさない): 出力契約(P1)・証拠チャネル(P0.0)・全ゲート。最も保護するのは出力契約(動かすと累積GT無効化)。
- 回すもの(螺旋): 単位(粒度・キー)と 基準(何を陽性とするか)。
- 単位スパイク・ループ(P0u): ①候補単位を置く ②分布が見える数で出力(本ビルドでなくスパイク) ③オーナーが反応
  ④俯瞰して診断 ⑤単位なら更新して再スパイク・基準なら記録して P4 へ(単位は触らない)。
- 反応の診断(載っている肝・局所では切れない): 単位 vs 基準は1反応ずつローカルに切れない。俯瞰=集約の形にしか現れない。
  - 俯瞰で見るもの: 分布の偏り / FP のクラスタ(誤りが共有する構造) / ラン横断の安定性(揺れるだけ=ノイズ)。
  - 診断: FP が共有構造を持つ・グルーピングがおかしい→単位 / 散発的なルール際→基準 / 揺れるだけ→ノイズ。
  - 例: 「別種を合計するな」は分布全体を見て・「期間が短いほど怪しい」は全22口座の gap が二峰{密≤7/疎≥8}に割れるのを見て出た。1例では見えない。
  - 一部は両義(「期間が短いほど」=keying変更)。俯瞰は曖昧さを減らすが消さない。残る両義例は迷ったら単位側＋オーナー判断。
  - 診断は自動分類器でなく、集約ビュー上のオーナー・イン・ザ・ループ。
- 俯瞰計器＝診断モード(新要素・G7 の例外・早く立てる): 開発者が全部を一望し診断する。review-server の同一サーバーのモードであって P2 GT作成モードと別ビルドではない=こちらは見る・突き合わせる、あちらは答えを作る。
  - 3ペイン・2ズーム: 詳細(左=ユニット一覧 / 中央=入力本文＋根拠を全ハイライト＋クリックでジャンプ / 右=judge 並置: 提案GT と 本番judge を横並び・各 verdict＋reason＋根拠リンク) と 俯瞰(出力分布＋誤りクラスタ＋ラン横断 flutter)。
  - 二重の仕事: ①俯瞰で単位vs基準を診断 ②提案GT と 本番judge の並置で食い違いケース=境界=GT作成モードへ最優先で送るべきケースが一目で分かる(食い違いが GT 投入を駆動)。
  - 無いと螺旋が収束せず診断自体が揺れの源になる。原型=あかつきの viewer 群(cog_xcall 生全文＋ハイライト・3ペイン・ab_review_server before/after)=畳むべき sprawl。昇格形=1サーバー設定駆動で judge を提案GT＋本番judge 両方並置。review-server スキルが雛形・ゲート側(S1–S12)。
- 終了条件: K回連続でオーナー反応が基準のみ(keying変更なし)→単位安定とみなし本決め。ここで初めてGTを作り P4 へ投資。
  ※SETTLE は勘でなく構造化反応ログから計算する: 各反応を round 付き・診断タグ(unit-keying/criteria/noise)で残し、unit-keying タグが K round 連続ゼロ=SETTLE。あかつきは未形式化(decisions 散文＋viewer 目視)で SETTLE は暗黙だった=ここが理想と実装のギャップ。記録は spike/(review-server 診断モードが捕捉)に置き、GT(verdict)・台帳(ADR)と混ぜない。
- 不変則: 単位が安定するまで GT も基準作り込みもしない(単位変更は累積GT/基準を無効化=認知が長引いた核)。
  定義型はスパイク1回で安定(規制)・創発型は複数回(認知)。同じ機構で自己終了=素性ごとの別装置は不要。
