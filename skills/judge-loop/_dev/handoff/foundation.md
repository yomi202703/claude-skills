## なぜこの仕事が在るか(発端)

judge-loop は今、軸を1本しか持っていない: フェーズ軸(Scaffold→P0.0→P0u→P1→P2→P3→P4→GT-established)+ ゲート G1–G10。「ハンドルを誰が握るか」=自律度/権限の軸が無い。10セッション並列運用(人間=聖徳太子化)を地で行く中で、人間を各ステップのペーシングから外したい——だが judge-loop の背骨は「オーナー=審級」(G3「no-GT では如何なる judge もオラクルでない・唯一の報酬信号はオーナー反応」/ G10「書いた実行プロンプトは全文サインオフ後に投入」)。

解: ペーシング(各ステップを人間が律速する)と 審級(最終的に何が正しいかを人間が裁く)を分離する。自律度軸は人間をペーシングから外すが、審級からは外さない。これが軸全体の不変条件。

## 確定した設計 — 自律度軸 L1→L2→L3

フェーズ軸に直交する二次元目。3段はメニューでなく依存連鎖(下の段の出力が上の段の栄養になる)。

- L1 人間ペース(現状): 人間が律速かつ裁定者。各ステップで確認しながら進む。判定ループの既定。
- L2 自走+モデル間裁定: claude-main が裁定者、claude-sub と gemma が競合する生産者。main がどちらが正しいか・何がベストかを決めて、人間を待たずに進む。人間は各ステップのクリティカルパスから外れる。
  - 罠=裁定者バイアス: 裁定者(claude)が生産者の片方(claude-sub)を、同じ事前分布・同じ失敗モードゆえ構造的にひいきする。相関した誤り。
  - gemma の脚の役割: 「もう一つの意見」ではなく誤りを脱相関させるための脚。
  - 不変の配線: main は自分の taste でなく外部アンカー(L1 が人間コミットで作った rough-GT / review-server の anchoring firewall = 人間がコミットするまで機械出力を見せない構造)に照らして裁く。これが無いと「似た二人が三人目に頷くだけの装置」に堕ちる。
- L3 自己起案(コンサル役): 判定基準そのものを系が提案する。逆向き要件定義。SE(仕様通り作る=閉じてる=暗黙の正解=仕様 がある)に対し コンサル(何を作る価値があるかを決める=開いてる=正解が無い)。no-GT は judge-loop の本拠地そのものなので、L3 = judge-loop が自分の no-GT 方法論をメタの問い「そもそも何を判定すべきか」に向け直したもの。
  - スコープ: no-GT のみ。正解がある系には不要(やらない)。
  - 提案のみ。ratification 抜きの自己コミットはしない(G3/G10 無傷)。L3 が破ってよいのはペーシングだけで、審級ではない。

依存連鎖: L1 が人間コミットでアンカーを作る → それが L2 の裁定基準になる → L2 が吐く override データ(人間が機械裁定を覆した記録)が L3 の採掘対象になる。順番に意味がある。

## L3 コンサル役の性質(grill で潰した核心)

「コンテキスト無関係な外部者」から出発して、二度言い直して確定した形:

- priors free / evidence bound: 案件の事前分布(内部の前提・政治・サンクコスト・なぜそう判定したかの内部解釈)からは隔離する——閉じたシステムの局所最適から出るため。だが案件の証拠(実際の出力・実際の取りこぼし・ユーザーが実際にやっていること、例: 音声md から「問題っぽい何か」)には接地する。データを一切読まないコンサル=占い/ホロスコープ。価値は無知でなく境界の把握。
- 境界認識・steerable: 「無関係な外部者」ではなく「要件を知っているからこそ要件の外を探索する特化」。仕様を知り、その補集合を探索する。仕様からの距離(隣接〜遠方)は固定でなく、メインセッションが起動時に振れる steerable なダイヤル。「もっと外を見ろ」「今は隣接でいい」を会話で指示できる。明示的に「外のみ」である必要はない。
- 探索と製錬の分離(これが最重要・刈るな): 
  - 探索(上流)= サブエージェントを fan out して刈らずに広く。ノイズ歓迎。「原石はノイズからしか生まれない」——探索を効率で刈った瞬間に街灯の下しか探さなくなる(G9 street-light bias の生成側での再演)。「案件によっては役に立たない」=鉱石の大半は不毛、その代償を受け入れて稀な一個を取りに行く探索のコスト構造。
  - 製錬(コンサル内部)= ノイズを smelt して圧縮する。これはコンサルが内部でやる。
  - schema が縛るのは出口だけ。探索の広さは一切縛らない。
- 出口スキーマ(固定): 「提案軸(こういう判定軸を持つべきでは)+ それを動機づける実ケースのクラスタ(証拠)」の対。生インスタンスだけ=ただのフラグ増(judge-loop 既存ゲート「One systemic finding ≠ N per-item flags」に反する)。生の軸提案だけ=ホロスコープ。両方揃って初めてコンサル。これは L2 の override ログ採掘と同じ形の一般化。

## judge-loop 内での位置づけ

- 自律度軸は SKILL.md に書き込まれる(フェーズ軸はそのまま、二次元目として追加)。
- L3 のコンサル役は別スキル `consultant` として起こし、judge-loop が routing する先になる(gemma-prompt / review-server / claude-md / task-handoff と同じ Composition の位置)。judge-loop は薄いオーケストレータの方針を守り、コンサルの中身は持たない。
- consultant は単独で立つスキル(オーナー裁定 2026-06-24「単独でも役に立つ」)。judge-loop の専用部品でなく、任意の no-GT データに対し「今見ていない死角」を提案する汎用スキルで、judge-loop L3 は呼び手の一人。description は judge-loop 非依存で自立。
- 理由: judge-loop は標準(スキル)であって実装でない。かつ TODO の build 系は全部「別ドメイン着手時」で Deferred(STATUS の overfit リスク回避)。今切るのは標準の確定であって、特定案件向けの動くコンサルを建てる仕事ではない。「gem-from-noise は閉じた仕様で書けない」=書けるのは規律(探索は刈るな/製錬は内部/出口だけ固定/不毛ケース織り込み)だけ、という事実自体が、これを機能仕様でなく標準の一段として書くべき根拠。

- G3: production judge は報酬でない。L2 の裁定者を「オラクル」に格上げしない。外部アンカー照合は measurement であって target でない。
