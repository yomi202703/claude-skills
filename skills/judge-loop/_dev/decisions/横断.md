## 2026-06-23 — G7 を単位依存度の gate 型問いへ書き直し(grill-me で詰めた結果)
- 発端: 「共通基盤(ディレクトリ・取り込み)は要件定義より前では」という問いから、現 G7 の内部矛盾が露見=「OTHER 横断を全部 SETTLE まで遅らせる」が、P0.0 が早期に命じる取り込み(pdf-to-md/xlsx)まで遅らせてしまう。
- 詰めの帰結(grill-me): 当初「3バケツ taxonomy を書き、別ドメインで検証」案を立てたが撤回。「何が単位依存か」は螺旋でしか分からない=紙で事前列挙する taxonomy 化は、単位を計器なしに紙で固める category error の再演(オーナー指摘「B=ロジック改善ループと同義」)。書けるのは早見表でなく gate 型の問いだけ。
- 決定: G7 を二分法(計器=早/他=遅)から「単位依存度で時を決める」へ書き直す。単位非依存(scaffold・取り込み・アクセス層)=早くてよい(出戻りなし)/単位依存=SETTLE まで/例外=診断計器は単位依存だが config 駆動で早く(再設定で吸収)。判別不能なら pre-SETTLE とみなし計器以外を遅らせる。
- なぜ今やってよいか(n=1 リスク無し): これは あかつきからの帰納でなく既存前提(単位は創発・動く土台に建てるな)からの演繹。ゆえに別ドメイン待ちで寝かせる必要がない。pressure-test 対象でもない。
- 観測変化(本丸): 次ドメインで「何を要件前に作り何を SETTLE まで待つか」の判断ミスを減らし、メタ作業の回り道を削ってロジック改善ループに時間を回す(方法論 §1)。
- 反映: SKILL.md G7 / docs/方法論 §5 G7 を両方書き直し(整合維持)。
