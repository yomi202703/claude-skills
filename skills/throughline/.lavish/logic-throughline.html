設計の原則

線は一本に絞った
仕組みを端から端まで貫く１本だけを描く。辿った経緯が要る時は記録を見る。
背景
元は「どう辿り着いたか（流れ）」と「今どう動く」の２枚だった。使ってみた持ち主が「流れの方は全然見ない、今どう動くしか見ない」と言った。流れは経緯の記録と二重持ちでもあった。だから流れを畳み、今は１本だけにした。
言葉は二層に分ける
説明の地の文は前提の無い人に分かる平易語。だが仕組みを動かしている指示文は無改変で全文引用（これだけが例外）。
背景
使ってみた持ち主が「ロジックの中にプロンプト全文が絶対欲しい」と言った。だが説明文は平易なまま保ちたい。プロンプトは言い換えると別物になる（正確な文言こそが挙動）ので、地の文は平易・プロンプトは無改変全文、と層を分けて両立させた。逐語にするのはプロンプト本文だけ（ルールや設定は要約のまま）。
一枚・自己完結
外部依存なし・単一ページ。見た目（描画）には触れず、中身だけ差し替えて更新する。
背景
前は出力を２ページに分け、それを左右に束ねる土台ページも要る作りだった。だが束ねる面の存在が混乱の元になり、その上片方のページ自体が読まれなかった。１枚に畳んで外部依存も無くしたら、束ねる土台ごと不要になった。
窓であって正本でない
中身の正はコードと決定の記録の側。これは生成した窓で、古くなったら捨てて焼き直す。
背景
真実がコードと記録の側にある以上、ページを常に追従させる仕掛けは二重管理になる。だから自動更新は作らず、元ネタの版とズレたら「古い」と出して人が焼き直す、に留めた。
出力の家は決め打ち
焼いた１枚は常に同じ定位置（人が読む常設の面）に置く。無ければその場所を作って導入する — 置き場を毎回選ばない。
背景
以前は定位置が無ければ別のフォルダや作業場所の直下に逃がす作りだった。だが散らばって「どこに出たか」が毎回ブレる。持ち主の「定位置を必須にして、無くても導入してほしい」に従い、常に同じ家に置く・無ければ作る、に変えた。

# throughline

## 出力(1枚を最新化する)

- 問い = 今どう動くか。源 = 今のコード/設計/プロンプト(台帳ではなくコードを読んで起こす)。形 = データの流れ(源→加工→出力→受け手)+ 設計原則 + 駆動プロンプトの全文。
- template = templates/logic.html。お手本 = `.lavish/logic-throughline.html`(throughline 自身の埋め済み view)。
- view≠真実(生成物)・単一HTML・file:// で開ける・描画には触れない。更新 = ソースを見て作り直し上書き。

## 作り方

骨組み templates/logic.html をコピーして埋める。密度・声・トグル/逐語の見え方は満タンのお手本 `.lavish/logic-throughline.html` を見る ── お手本は丸ごとコピーしない(中身が固有で残留する)。

1. 今のコード/設計を読み、「この仕組みは今こう動く」を html-deck の規律で削って1枚にする。状態の羅列でなく、仕組み(データの流れ・構造・原則)を描く。
2. 仕組みを駆動するプロンプトファイルがあれば、その本文を逐語で全文埋め込む。要約しない(言い換えると別物になる)。複数あれば、どの段を駆動するかを添えて各々全文。プロンプト以外の源(ルール/spec/設定)は平易に要約する ── 逐語にするのはプロンプト本文だけ。
3. 本文(導入・データの流れの段・原則カード〔背景トグル任意〕・プロンプト全文ブロック)を差し替える。描画(CSS)には触れない。
4. ある決まりが「過去にこう決めたから今こう」である時、その背景を畳んだトグル(JS 不要の素の `<details>`・既定は閉)で添える。中身は経緯の記録から平易語で起こす。これは今の選択への「なぜ?」であって時系列の物語ではない。背景の無い決まりには付けない。
5. 鮮度 = ソース(コード/プロンプト)のハッシュをページに刻む。今のハッシュと違えば「古い」と表示して作り直す。時刻でなく中身のハッシュで判定。

## language ルール

- 説明の散文は内部語を出さない。メタ語(spine/DAG/ledger 等)、id、スキーマ語、ツールの固有名、できる限りファイル名を本文に入れない。読む人が知らない言葉はゼロ、一般的なエンジニアが分かる語だけ。
- 例外はプロンプト全文ブロック。逐語の証拠なので無改変で載せる(内部語・ツール名・スキーマ語が入ってよい)。機構そのものの引用だと分かるよう囲い、説明の散文と逐語ブロックの二層を混ぜない。
- 実装の正はコード/プロンプトファイルにあり、これは生成した説明だと分かるように書く。
- 合否: その分野の前提が無い人が散文で今の動き方を掴め、かつ駆動プロンプトを一字一句そのまま読めるか。どちらか欠ければ書き直す。

## 出力先

- 家は常に `.lavish/`。`.lavish/logic-<slug>.html` に上書きし、`.lavish/` が無ければ作る(別所に散らさない)。場所は repo のルート、repo-shape がスロットを定めていればそれに従う。
- `handoffs/` 下には置かない ── そこは handoff の契約の領分、ここは人が読む派生ビュー。
- slug は仕組みの名から引いた短い英数。file:// で開けば動く(外部依存なし)。
- 置き場と firewall 分類は repo-shape が定める。`.lavish/` の view は再生成物だが意図的に commit する。ここはビューの中身だけを作り、スロット配置は repo-shape に委ねる。
