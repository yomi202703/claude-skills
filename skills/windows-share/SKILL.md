# windows-share

Mac 由来の成果物を Windows へ渡す前の、送り手側・静的・毎共有のプリフライト監査＋修正。
correctness バグ探しでも新規クロスプラット対応の生成でもない。既存対応の 乖離・漏れ・欠落 を共有前にゼロにする。

固定パイプライン: 0 共有セット確定 → 1 検出(6軸) → 2 doctor 点検 → 3 分類と修正 → 4 検証 → 5 報告。

## 原則

- W1. 送り手側・静的に徹する。「Windows で動くか」は実機がないと検証できないので検証しない。runtime の pass/fail は受領側 `doctor` に寄せ、スキルはその Windows カバレッジを点検するに留める。VM は立てない。
- W2. 共有セットは案件ごとにその場で確定。規約ディレクトリで固定しない。読めなければ尋ねて止まる。不変の除外は git履歴 / `.venv` / cache / `.DS_Store` のみ。
- W3. 共有物は3バケツ。混同しない。
  - 必須配布物(無いと動かない): `.env`・`_data` の seed・起動スクリプト・成果物。「在るか・移植可能か」を保証。欠落は往復の最大要因。
  - ノイズ(絶対に渡らない): git履歴 / `.venv` / cache / `.DS_Store`。除外し、紛れていたら漏洩として検出。
  - 秘密: `.env` は秘密だが Windows 開発者には必須。「渡さない」ではなく「正しく渡ったか・本文が移植可能か」を見る。
- W4. ツール出力(`file`/`xxd`/`grep`)は候補。最終判断は開いて読む。符号化の正解は「種別 × Windows で何が開くか」で変わる(下表)。
- W5. 修正は確信ゲート。一意に直せるものだけ直す。意図依存で割れるもの(どのコマンドが正か・手順の意味・符号化方針・配布経路)は直さず提示。
- W6. 直した全件を file:line ＋ 根拠で報告に残す。確認できなければ「確認できなかった」と明記し捏造しない。

## 符号化ポリシ表(種別 × 開くもの → 正解)

- `.bat` → cmd.exe: BOM無し UTF-8 ＋ 先頭 `chcp 65001` / `set PYTHONUTF8=1`、CRLF。BOM付きは旧 cmd が先頭行で死ぬので禁止。
- `.ps1` → PowerShell: BOM付き UTF-8(PS5.1 は BOM無しを CP932 誤認)、CRLF。
- `.md` `.txt` → メモ帳/サクラ/VSCode: UTF-8。CP932 既定エディタ運用なら BOM付きが安全側(方針割れ→提示)。
- `.html` → ブラウザ: `<meta charset="utf-8">` 必須、BOM 不要。
- `.csv` → Excel: BOM付き UTF-8(無いと CP932 誤認で化ける)か CP932。
- `.tsv` → スクリプト読込: UTF-8 LF(`newline=""` 維持)。
- `.json` `.yaml` `.toml`: BOM無し UTF-8(BOM付きはパーサが死ぬ)。
- `.xlsx` `.db` 画像 音声: binary、変換禁止。`.gitattributes` に明記。

## Phase 0 — 共有セット確定

会話から「今回 Windows に渡すもの」を拾い確認する。規約で勝手に決めない。読めなければ尋ねて止まる。不変の除外を外し対象を固定。

## Phase 1 — 検出(共有セット内・6軸)

ヒットは開いて読む。`<軸ID> → file:line / 内容`。

- A1 欠落(配布物): `.env`(or `.env.example`＋取得手順) / seed / 起動スクリプトが揃うか。`.command`/`.sh` に対応する `.bat`/`.ps1` の片割れが在り同期しているか。
- A2 パス: 絶対/ホーム依存 `/Users/` `~/` `c:/Users/...` `file:///...`。
- A3 コマンド: mac専用語 `open` `pbcopy` `brew` `python3`限定 `sed -i ''` 等 BSD フラグ。
- A4 符号化: 各ファイルを上表に照合(`file`/`xxd` で BOM、HTML は `charset`)。
- A5 改行: テキスト LF＋`.gitattributes`、`.bat`/`.ps1` は CRLF か。zip/Drive 共有は `.gitattributes` が効かないので実バイトを確認。
- A6 欠落(手順): README 等に Windows 起動/セットアップ手順が在り AI 無しで辿れるか。

## Phase 2 — doctor 点検

受領側 `doctor` が在れば、A1 の必須項目を自己診断でカバーしているか点検し穴を提案。無ければ「即時自己診断が無い」を欠落として報告。

## Phase 3 — 分類と修正(確信ゲート)

- 確信あり(一意): パス除去/相対化、BOM/改行正規化、HTML `charset` 付与、CSV BOM 付与、`.bat`↔`.command` 片割れ生成・同期、`.gitattributes` 補完、ノイズ除去。→ 直す。
- 要判断(意図依存): どのコマンドが正か、手順文、`.md` の BOM 方針、`.env` の配布経路、seed の配布要否。→ 提示。

会話で方針確定済みなら要判断→確信ありに格上げ(根拠を W6 に明記)。最小・局所に。

## Phase 4 — 検証

テスト/型/lint が在れば実行、退行は revert→要判断へ降格。符号化/改行は実バイト再確認、HTML は `charset` 再 grep。無ければ再読・目視と明記(緑を主張しない)。

## Phase 5 — 報告

```
## 共有セット   対象: <群> ／ 除外: git履歴/.venv/cache/.DS_Store
## 欠落(必須)   A1: file:line / 何が無い・片割れ乖離（往復の主因）
## 直した       F1: file:line ← A2 / 根拠
## 要判断       J1: file:line / 何が割れるか・選択肢
## doctor       カバー済み / 穴: <項目>
## 検証         テスト/型: pass/fail（無→目視）／符号化・改行: 再確認 OK
```

## 終了条件 / 失敗時

6軸走査・確信あり適用・検証緑(退行は降格)・要判断/doctor穴/必須欠落を提示で終了。
共有セットが読めない→尋ねて停止。在処不明→「確認できなかった」と明記。実機可否→検証していない旨を明記し doctor に委ねる。
