# html-deck

クライアント向けの HTML レポート/デッキ(判定・スコアリング結果の表、口座/項目一覧、基準表など)を作る・レビューするときの作法。中核は一つ:

- 自明になるまで削り、判断は数字の見せ方で表す。
- 全要素に存在理由を要求する。証明できないものは消す。

これはロジック設計ではなく見せ方の規律。判定の中身は judge-loop 等に委ね、ここは「出来た判定を意思決定者にどう見せるか」だけを扱う。

## 6原則(それぞれ do / don't)

1. 自明になるまで削る
- 他で言っていない・推測できない情報だけ残す。見出しは冗長な括弧書きを剥がして名詞だけに。
- 同じ情報を2度出さない(基準を上で具体化したら下の重複表は消す。49行表が2本あったら1本に)。
- don't: 注釈・小見出し・「結論/前提/流れ」のような説明ブロックを足し続ける。

2. ラベルより生数値。判断は強調で表す
- verdict を別カラムの文字(要確認/○)で持たず、該当セル(回転率・手数料化率など)を閾値超で着色する。複数軸が全部色＝該当、という読み方にする。
- 比率は実数(%)で出し、該当だけ色を変える(全部に色を付けない)。
- 全件出す＋少数を強調(「隠さないが見るべきはこれだけ」)。
- 強調は色だけに載せない。色を唯一の意味担体にすると印刷モノクロ・色覚多様性で判断が消える(@media print を謳うなら自己矛盾)。太字・セル左ライン・位置で二重符号化し、色が落ちても該当が読めるようにする(雛形 template.html は太字＋左の赤ライン＋背景色で既にこれを満たす)。
- don't: verdict ラベル列と生数値を二重に持つ。チップ/タグで定性表示して数字を隠す。色だけで判断を表し、モノクロにすると意味が消える。

3. 意思決定に効かない列・行は落とす
- 各列に「これは判定/優先順位を動かすか」を問う。判定非依存・常に同値・不透明な内部コードは消す。
- 例の失敗: 常に2つ並ぶ「あてはまった項目」(情報量ゼロ)、判定に使わない内部ランク(I/D/G)を凡例なしで、過剰検知の参考件数。
- don't: 「一応あるから」で列を残す。

4. 並びが優先順位を語る
- 列順＝判定の隣にその根拠指標(判定↔それを決めた数値、別軸↔その分母)。
- 行順＝要確認/該当を上に固め、その中は重い順(主要指標の降順)。上から読めば優先順位になるように。
- don't: 主キーや単純昇順のまま放置して、該当が非該当の下に埋もれる。

5. 相手の言葉で書く。記号・内部表現を排す
- 元データの正式表現を使う。内部の符号化(「行1/行2」等)は出さない。
- 数学記号(∧ ≥)・社内略語(「手化」)・専門用語をそのまま客先に出さない。自然な文に開く(「両方が超えたとき」「以上」)。
- don't: 開発者の頭の中の表記をそのまま貼る。

6. 過剰主張しない・氾濫させない
- 数値表現が断定に聞こえないか(「上位N割」は「N割が違反」に聞こえる、「違反がありそう」は言い過ぎ)。意味ベースの中立表現に。
- 該当(要確認)は出しすぎない。閾値は高めに1本、本当に突出した数件だけ。氾濫したリストは無視される(precision 優先)。
- don't: recall を上げて要確認まみれにする。2段階で過剰なラベルを付ける。

## メタ原則(最重要)

- 守りの注釈は設計の失敗の裏返し。「これは違反ではありません」式の免責を足したくなったら、まず現物を直す ── 誤読を生んでいる名前・判定ラベル列・分母・並びを変えて、注釈が要らない状態にする。免責文を消せたら勝ち。
- ただし「消せたら勝ち」は誤読を糊塗する注釈に限る。次の2つは守りの注釈ではなく構造情報で、消さずに現物の一部として見せる: (a) 分母・期間・閾値とその根拠(その数字が何分の何で、なぜその閾値か)。(b) 法定・契約上必須の開示(規制領域では注釈が義務のことがある)。両者を「免責だから削る」と混同しない。
- 削るのは受け手の理解が上がるときだけ。削りは送り手の自信であって受け手の理解ではない ── 前提を共有しない読み手には、冗長さや確認情報がむしろ信頼の源になる。削りすぎは信頼を削る。同じHTMLを複数の読み手に出すなら「効かない列」は相手依存になる(万能デッキ前提を疑う)。
- 判定条件(「自明まで削る」と「削りすぎは信頼を削る」のどちらが勝つか): 著者が頭の中に持つ・他から導出できる・既出の繰り返し → 削る。読み手が照合や次の行動に要る物(生数値・分母・閾値根拠・保留理由・法定開示) → 残す。削ると検証ができなくなる物を削ったら「削りすぎ」。

## 作る順(author)

- データ先、装飾後。まず全件の生数値表を出し、そこから削る/強調する。
- 大量行は手で書かない。スクリプトで算出→HTMLを生成し、既存ファイルへ正規表現で差し替える(49行を手編集しない)。
- 単一 HTML(インライン CSS/JS)。UTF-8。print 対応(@media print)。色は意味を持つときだけ。

## 雛形(templates / scripts)

- 推奨雛形は templates/template.html。見た目は作り込む(濃色ヘッダー・罫線・角丸・カード影・メリハリ)が、情報設計だけ規律を効かせた最小デッキ。装飾を削れという意味ではない ── 削るのは死に列・verdictラベル・免責注釈であって、見た目ではない。
- scripts/gen_template.py がその生成元(データ先・スクリプトで算出→HTML)。rows と閾値を案件に差し替えて実行すれば template.html を上書き生成する。軸が2本でなければ列定義と該当判定を増減する。
- templates/antipattern.html は対の悪い版(verdict列の二重持ち・死に列・免責注釈・ID順)。review 時の「これをやっていないか」の具体物として参照する。
- templates/sample_judgment_table.html は実案件由来の正準サンプル(匿名化済み)。1ファイルで2系統を兼ねる: タブ1=判定表(下記「判定表の型」)、タブ2=抽出一覧(下記「抽出一覧の型」)。template.html の上位互換で、判定・抽出の一覧を作るときの到達目標として読む。

## 判定表の型(sample_judgment_table.html に学ぶ)

最小の template.html に無く、MTGに出せる資料が備える3つの型。判定表を作るならこれを足す。ただしこの型は意思決定層(結論を一意に近づける)向け ── 監査・反証・原因探索が主目的の層では、funnel/大数字先出しはむしろ邪魔になる。読み手の層で出し方を変える。サンプルは優れた一例であって規範ではない。

- funnel を背骨にする。各セクションの先頭に「全N → 該当a → 該当b → ○残り」の漏斗を大きな数字で置く。意思決定者は漏斗だけで結論を読み、その下の全件表は監査証跡。並びだけでなく、結論の見出しを数字で先に出す(削るだけでは足りない/次の行動が一意に近づく構造)。
- 閾値を単位ごとに可変にする。読み手や対象の属性で「効く線」が違うなら一律閾値にしない(例: 投資方針別に回転率の線を変える)。保守側を低い線で拾い、積極側を高い線で正しく外す。可変にした事実と、それで拾えた/外せた具体例を note に一言。これは原則6「閾値は高め1本」と矛盾しない ── 単位ごとに高い1本(precision は各単位で保つ)であって、線を増やして氾濫させることではない。
- 判定保留は件数を上・行を下。保留の存在と件数は funnel(結論の見出し)に出す ── 未確定は隠さない。そのうえで個々の保留行は偽の verdict を付けず「—」＋理由で監査表の最下段へ。保留自体が意思決定のブロッカーなら最上段で警告する(下段に沈めるのは「該当でも非該当でもない行」が優先順位を乱さないようにするためで、未確定を隠すためではない)。確定violationゼロや保留件数の明記は過剰主張より信頼される。限界は「相手が何を出せば解けるか」に翻訳する(免責で終わらせない)。

## 抽出一覧の型(sample_judgment_table.html タブ2に学ぶ)

同じ実物サンプルのタブ2(録音から説明済み項目を抽出し未充足を一覧化)が抽出一覧系統を兼ねる。判定表と共通の funnel・保留の作法に加え、抽出固有の3つ。

- 抽出物そのものを payload に出す。件数だけで終わらせず、抽出した中身(未充足項目=「流動性・税金・…」)を行内に並べる。読み手が次に当たるべき対象が一覧になる。
- 「原典に無い」を独立ステータスにする。source(録音)に見つからない=違反、にしない。not-found は「判定不能」等の別状態で持ち、確定違反と峻別する。non-finding を負の判定に格上げしない(でっち上げの違反を出さない=precision)。ただし探索範囲・検索努力(どの source をどこまで当てたか)を併記し、「原典に無い」を「探さなかった」と読み手が区別できるようにする ── そこを書かないと中立化が抜け穴になる。
- 土台データ行列で結論を裏付ける。抽出結論(該当0件)を、全件×分類の生カウント行列で支える。各セルは生の回数＋帯で着色(直近1件=赤/浅い=橙/厚い=緑/なし=灰)の二重符号化。全件見せて、突出だけ色で立てる。

## 視覚の作法(プロのデータテーブル規約)

判定特有の情報設計(上の各節)は当スキル固有。視覚の craft はプロのデータテーブル規約を借りる ── ただし参照先の多くは操作型ダッシュボードで、verdict表示の既定は当スキルと衝突する。craft だけ取り、既定は捨てる。

借りる:
- 数値は右揃え・テキスト左揃え。数値列は見出しも右揃え(桁が縦に揃って比較できる)。
- 数字は等幅数字(font-variant-numeric:tabular-nums)。桁幅が揃いスキャンできる。
- 色は3〜5色まで・意味を固定。コントラスト比4.5:1以上、色覚ツールで検証(色以外の担体は原則2の二重符号化で既述)。

借りない(ダッシュボードの既定だが当スキルと衝突):
- 信号機(赤黄緑)グラデを全行に塗る条件付き書式 → 全件着色は原則6の氾濫。閾値超の例外だけ塗る。
- 数値を置き換えるチップ/タグ/アイコンでの定性表示 → 生数値を隠す＝原則2違反。ただし数値に併せる記号・アイコンは可(形は色に依存しない担体＝原則2の二重符号化の一手段)。棄却するのは「置き換え」だけ。
- 密度トグル等のインタラクション前提 → 当デッキは印刷・単一HTMLの静的物。compact を選んで出す。

## 出荷前チェックリスト(review)

- この見出し/文は他で言っていないか・推測できないか → 削る
- verdict をラベル列で足していないか → 生数値＋強調で表せ
- 強調が色だけになっていないか → モノクロ印刷で該当が読めるか(太字・左ライン・位置で二重符号化)
- 閾値・分母・期間の出所が消えていないか → これは守りの注釈ではなく構造情報、見せる
- 各列に存在理由があるか(定数・判定非依存・不透明記号は落とす)
- 並びは優先順位を語っているか(列=判定の隣に根拠/行=該当を上・重い順)
- 記号(∧ ≥)・内部略語・内部コード(行N)が残っていないか → 相手の言葉に
- 注釈で守っていないか → 現物を直して注釈を消せないか
- 該当が多すぎないか(precision・閾値は高め)
- 数値表現が過剰主張していないか(「上位N割」「違反がありそう」等)
- 重複していないか(同じ基準表/集計が2箇所、49行表が2本)
- 結論の見出し(funnel等)が先にあるか → 全件表だけで意思決定者に読ませていないか
- 読み手で効く線が違うのに一律閾値になっていないか → 単位ごとに可変か
- 判定できない行に偽の verdict を付けていないか → 件数は funnel(上)・行は「—」＋理由で最下段、限界は相手のアクションに翻訳
- 数値は右揃え・等幅数字か/コントラスト4.5:1以上か(プロのデータテーブル規約)
- 信号機グラデを全行に塗っていないか → 閾値超の例外だけ着色

## 内部表現の匂い(原則5の検査)

地味だが一番効く。著者は内部マッピングを頭に持つのでコストを感じず、払うのは読み手だけ ── 善意では守れないので匂いで機械的に拾う。各見出し・行ラベル・凡例・脚注を1個ずつ当てる。

- 内部コード・ランク(I/D/G、Tier、区分記号)
- 行/列の位置参照(行1・行2・col3・上から3番目)
- スキーマ由来の名前(列名 v_hit、enum値、フラグ名、閾値の変数名)
- 開発略語・社内符号(「手化」、CER を説明なしで)
- 数学記号(∧ ∨ ≥ ≤ →)を客先にそのまま
- パイプライン用語(stage / judge / extract 等の処理段名をそのまま)
- 対処: それぞれ「相手が自分のデータとして照合できる正式表現」に開く(投資方針名・正式商品名・相手の口座番号など)。開けないなら、それは判定に使うべきでない内部物の疑い ── 列ごと落とすか再定義する。
