  - 閾値・分母の出所が消える/「免責注釈=失敗」は行き過ぎ(規制領域では法定注釈が必須): → メタ原則に例外を追加。(a)分母・期間・閾値根拠 (b)法定・契約上の開示 は守りの注釈でなく構造情報、消さない。
  - 「削る=普遍善」が一番危うい前提(2者一致): 削りは送り手の自信であって受け手の理解ではない。→ メタ原則に歯止めを一行(受け手の理解が上がるときだけ削る/万能デッキ前提を疑う)。

- 私(統合役)の補正: 「色一本足」批判はコードを読めば雛形に当たらない。レビュー陣はソースを読んでいない分そこを過大評価していた、と記録。

## 2026-06-27 「内部表現を排す(原則5)」を checkable 化

- ユーザー観察:「単純に、内部表現を避けるとかが地道に効く」。最も過小評価されやすいが最も効く原則という認識を共有。
- 根拠: 単発の勝ちが無く全セルに薄く効く/著者はコストを感じず読み手だけが払う非対称ゆえ善意では守れない/資料単独で正しく読まれる条件。
- 対応: SKILL.md に「内部表現の匂い(原則5の検査)」節を追加。内部コード・位置参照・スキーマ名・開発略語・数学記号・パイプライン用語の6匂いと、各々「相手が照合できる正式表現に開く/開けないなら落とす」対処。原則5とアンチパターンに散っていた検査を1か所へ集約。

## 2026-06-27 プロのデータテーブル規約を craft 層だけ取り込み(視覚の作法)

- ユーザー提起:「本物のデザイナーが作る html を web で参照した方が良いのでは」。妥当。ただし参照先の多くは操作型ダッシュボードで verdict 表示の既定(信号機グラデ・ステータスチップ)は当スキルと衝突するため、視覚 craft 層だけ取り情報設計の既定は捨てる、と線引き。
- web 調査で確認できた一致/補強: (1)数値右揃え・等幅数字(プロの定石、template.html は既に準拠) (2)コントラスト4.5:1・色覚検証・色以外の担体併用 ← 3者レビュー最重要の二重符号化を外部 a11y 規約が独立に要求(裏書き) (3)色3〜5・意味固定=原則6 (4)Tufte データインク比/冗長インクを消せ=原則1・2の源流。
- 衝突ゆえ「借りない」と明記: 全行への信号機グラデ(=氾濫)、アイコン/チップで生数値を隠す。
