# transposed — field-major レイアウト（行=項目, 列=レコード）

分類器が `transposed: true` を立てたシート用。**行と列の役割が通常表と逆**になっている。

```
        ┌ col B …       C      D      E   …  (レコード/サンプル)
 row1   │ ＃            1      2      3
 row2   │ サンプル名     名A    名B    名C
 row3   │ 項目X         値     値     値        ← 各行が1フィールド
 row4   │ 項目Y         値     値     値
   ⋮
```

## なぜ専用扱いが要るか

`header_row_index` / `header_preview` は**信用しない**。通常表前提のヘッダ検出は転置表で無意味な行（データ行の1つ）を拾う。`transpose_confidence`（row方向の型一貫性 − col方向の型一貫性）が正で大きいほど確度が高い。

## 読み解き手順

1. **ラベル列を特定**: 先頭1〜2列（`label_cols` 相当）が項目名スパイン。A列が疎な区分ラベル、B列が項目名、という二層もよくある（A列はマージ無しでも下方向 forward-fill）。
2. **ヘッダ行を特定**: 上端1〜2行がレコード識別子（番号・サンプル名・ID等）。これが列C以降の各レコードを定義する。
3. **本体**: 残りの各行が1フィールド。列C以降がレコードごとの値。
4. **型の手掛かり**: 各行は型が揃い（例: 郵便番号行は全て数値、日付行は全て日付）、列方向は型がバラつく。これが転置の証拠。

## 出力グレイン（content_type 既定を上書き）

転置表では「1行=1レコード」ではなく「1行=1フィールド」。素直に行を流すと壊れる。推奨:

- **field-major JSONL**（既定）: 1行目に `_type:meta`（レコード凡例 = ヘッダ行のID群、列定義、適用したマージ・吹き出し）、以降は1フィールド=1レコード:
  ```json
  {"_type":"field","行":8,"区分":"意向確認書","項目名":"保険契約者_郵便番号",
   "レコード":{"1":4530014,"2":5530004,"…":"…"},"備考":null,"訂正":"正：5520011"}
  ```
  - `レコード` はヘッダ行のID（`＃` や サンプル名）をキーに。
  - 右端の注釈列（備考・訂正・確認結果など）はフィールド単位で各レコードに添える。
- レコード数が少なく素直に縦持ち化したい場合のみ、tidy/long 形式（1セル=1行: `{レコード, 項目名, 値}`）も可。判断は無確認で。

## 生成スクリプト（P4 経路の場合）

`p4_structured_large.md` に従い、転置を機械的に解く idempotent スクリプトを書く:
- ヘッダ行（レコードID）とラベル列（項目名）をハードコードせず `header_rows`/`label_cols` から解決。
- 日付・シリアル値は `merges.md`「日付シリアル値」節に従い正規化。
- 出力件数 = フィールド行数（`data_rows[1] - data_rows[0] + 1`）を assert。

## マージ・吹き出し

通常どおり `merges.md` を適用。転置表は右端に「備考 / 訂正 / 確認結果」列を持ちやすく、その意味づけが浮動テキストボックスに書かれていることが多い（`drawings.md`）。吹き出しテキストは列セマンティクスの定義として meta に保持する。

## 注意

- 色（赤字/青字・セル塗り）で「どのレコードが誤りか」を示す運用がある。色情報はセル値から取れないため JSONL には乗らない。必要なら `p6_visual.md` か原本目視で確認し、その旨を出力の注記に残す。
